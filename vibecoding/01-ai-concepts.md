你手机里大概率装了一个 AI 应用——可能是 ChatGPT，可能是 Kimi，可能是 DeepSeek，也可能你用的某个 App 里悄悄塞了个 AI 助手。你跟它聊天、让它帮你写东西、翻译、总结，它什么都能聊上两句。

但你有没有想过：它到底是什么？为什么它能回答几乎所有问题？为什么有时候它又一本正经地胡说八道？为什么有的 AI 只能聊天，有的却真的能帮你干活？Agent、RAG、MCP、Function Calling……这些词你可能在哪听过，但一直没搞明白。

其实大模型没那么神秘。要理解它，最简单的方式就是把它想象成一个人的大脑——一个脱离了身体的、光秃秃的大脑。这个大脑只能做一件事：接受信息 -> 思考 -> 输出。记住，这是大模型最核心的逻辑。

## 01 光秃秃的大脑

市面上有很多大模型，像 Gemini 3.5 Flash、GPT 5.5、Claude 4.7、DeepSeek V4-Pro、GLM 5.1、Kimi 2.6 等等都是当前各家最新最强的模型。既然把它们当作大脑来看，每个大脑的智力自然不一样，也各有自己的性格特点。当然，随着技术进步，到现在各家模型的差距已经没有一开始那么大了，但仍然各有各的强项。不过有一点可以确定——它们都比几年前聪明太多了。

这些大脑学习了互联网上海量的文本和数据，可谓是上知天文下知地理，比起你我来说，肯定要懂得更多，知识储备更丰富。

也就是说，当你打开一个 AI 应用，在输入框里打字的时候，跟你对话的就是这样一个大脑。你可以把自己任何想问的问题告诉它，它会回答你。你问它问题的方式，就是我们常说的**「提示词」（Prompt）**。你跟它说的每一句话，其实都是在写提示词——你说得越清楚，它回答得越好；你说得含糊，它可能就跑偏了。

> 提示词 Prompt：你跟大模型说的每一句话，都是提示词。一开始业内搞「提示词工程」，甚至专门招 prompt 工程师，是因为大脑还不够聪明，只能说清楚一点。到现在只要你能把事情说清楚，模型一般都能很好地理解并执行。

值得注意的是，因为它只是个大脑，所以它只能跟你交流、回答你问题，没法给你做更多的事情。比如你想做个网页，它没法帮你打开 IDE 写代码，更没法帮你部署上线。

总不能就让这么聪明的一个大脑天天只跟你聊天吧？你可以想象自己从市场上招聘了一位学识丰富的博士，但他现在只能沟通没法直接给你干活，你需要对他进行一定程度的改造，让他变得有生产力。于是故事开始了——

## 02 装上眼睛和耳朵

你首先给他装上眼睛，让他能看到你和这个世界。

**改造前**：你发一张花的照片问他「这花好看吗？」，他说：我看不到图片，只能处理文字。

**改造后**：你发同一张照片，他直接告诉你：这是绣球花，蓝紫色的品种，拍照的光线是从左侧打过来的，构图不错，就是背景稍微有点杂。

但他还是听不到声音——你说什么他听不见，你们只能文字沟通，这很麻烦。所以你又给他装上了耳朵，这下他能听到你说话了，事情逐渐变得酷起来：他可以跟你打视频电话了，能看到你、还能听到你在说什么，开始像个人了。

> 多模态 Multimodal：能看能听，就是大模型的多模态能力。眼睛对应视觉理解（图片 / 视频），耳朵对应语音理解。

## 03 装上记忆

能沟通了，看得见了，也听得着了，但是你逐渐发现了一个问题：他有点健忘！刚说的事情，转头他就忘了。比如你上午告诉他你喜欢喝美式，下午问他推荐什么咖啡，他说来杯拿铁吧；昨天跟他聊的项目方案，今天想接着聊，他一脸茫然。

你一番研究，发现原来他缺了一样关键的东西——记忆。于是给他装上了记忆。

你给他装上的第一种记忆叫短期记忆，就是你跟他当前这次对话的聊天记录。大模型的上下文窗口（Context）就是这个短期记忆的容量——你每次跟他聊天，他都在努力记住当前对话里说的每一句话。聊得越多，需要记住的内容就越多，逐渐地上下文就满了，他就会开始忘记前面聊的事情，这时候你就该另起一个话题了。现在强大的模型，短期记忆已经达到了百万级别的水平，能记住相当长的对话。

> Token —— AI 世界的计价单位：大模型理解文字的方式跟我们不一样，它不是按「一个字」来处理的，而是把文字切成一个个小片段，每个片段就是一个 Token。一个汉字大概就是一个 Token，一个英文单词可能是 1-2 个 Token。上下文窗口的容量就是用 Token 来衡量的——「百万级别的上下文」就是能装下大约一百万个 Token。你在 AI 产品里看到的「本次消耗 XX Token」，也是这个意思，Token 就是大模型世界的计价单位，聊天越长、记忆越多，消耗的 Token 就越多，花费也就越高。

继续回到记忆的话题。你给他装上了短期记忆，以为这下总该没问题了吧。不过，就在你以为一切顺利的时候，你发现了一个让人头疼的问题——

## 04 他有时候会一本正经胡说八道

你问他一个不存在的论文，他能给你编出作者、期刊、发表年份，看起来像模像样，但全是假的；你问他一家餐厅的地址，他可能给你编一个看起来特别合理的地址，但根本不存在。

> 幻觉 Hallucination：这叫幻觉，是大模型最让人头疼的毛病之一。为什么会产生幻觉？因为大模型回答问题的方式，不是「查到什么说什么」，而是根据概率猜下一个字是什么。就像一个特别会考试的学生，遇到不会的题也不会空着，他会根据经验编一个看起来最合理的答案。大模型也是这样——它不知道自己不知道什么，所以当它缺乏相关知识的时候，它不会说「我不知道」，而是会编出一个看起来合理的回答。

那怎么对付幻觉呢？办法就是让他**「有据可查」**。你给他接入了一个知识库——当你问他问题的时候，他会先从知识库里检索出相关的内容，然后再结合检索到的信息来回答你，而不是凭空编造。这个机制叫做 RAG（检索增强生成）。有了 RAG，他的回答就有了依据，幻觉的问题大大缓解了。

> RAG 检索增强生成：RAG 不止解决了幻觉的问题，它还顺便给了大模型另一种记忆——长期记忆。你以前跟他说过的事情、你的个人偏好、你的工作背景，都可以存进知识库里。这样他不仅不会胡编，还能记得你的个人喜好、记得你之前交代过的事情，你会有种错觉——自己好像跟他很熟，他好像也很了解你。

**记忆是大模型的核心。** 失忆的大模型是个残废，记性好的大模型则让你事半功倍。

## 05 装上双手和工具箱

回到正题。从现在开始，他开始像个正常人了，能正常地跟你沟通。但是你发现，虽然他很聪明，来来去去还是只能跟你聊聊天，别的啥也帮不上。于是你灵机一动——给他装上一双手，还配了一个专属的工具箱。

双手是他调用工具的能力，工具箱里则是各种各样具体的工具，像网页搜索、PPT 生成、代码部署等等。双手和工具箱是一体的：光有手没有工具干不了活，光有工具没有手也用不起来，必须一起装上才行。

**改造前**：「帮我查一下明天的天气。」->「抱歉，我无法访问互联网，无法为你查询实时天气。」

**改造后**：「帮我查一下明天的天气。」-> 他自动调用了网页搜索工具，几秒后回复：明天北京多云转晴，最高 28°C，最低 19°C，适合出行。

有了双手和工具箱，大模型终于可以真正干活了——它可以在需要的时候使用网页搜索工具去寻找自己需要的资料，可以使用 PPT 生成工具来做 PPT，它开始有产出了。

> Function Calling + API / MCP：大模型的双手就是它调用工具的能力（Function Calling），而工具箱就是常说的 API / MCP。API 是「应用程序接口」，是工具调用的通道；MCP 是「模型上下文协议」，是更标准化的工具接入方式。

## 06 教他拆任务——Agent 登场

有一天，你安排了一个任务给他，让他帮你回一下邮箱里的未读邮件。你发现这么简单的事情，他居然不会！你问他为什么，他说「不知道从哪开始」。你思考了很久，原来是这个任务的链路太长了——他现在还只会做单一的任务，对于多步骤的任务还没学过。

于是你开始教他：接到任务的时候怎么理解目标，怎么拆分完成路径形成一条条待办，想清楚每个环节需要调用哪些工具，然后按待办去执行，执行过程中还要检查任务有没有完成、是否符合要求，完成了就标记一下进入下一个，直到最终目标完成。

果然，改造之后的他顺利地完成了任务。到这里，他已经有了 Agent 的雏形——他会思考，会自主规划和执行任务。这个**「思考—行动—观察—再思考」**的循环过程，就是最经典的一种 Agent 框架，叫 ReAct（Reasoning + Acting）。

当然，Agent 框架不止这一种。不同的框架，就像不同的工作习惯：

- **ReAct**：想一步、做一步、看一步，再想下一步——适合需要灵活应对的场景，比如处理邮件、搜索资料。
- **Plan-and-Execute**：先把整件事的步骤全规划好，然后一口气执行到底——适合目标明确、步骤清晰的场景，比如按固定流程生成报告，一般编程 Agent 就是这种。
- **Reflexion**：做完之后还会回头审视自己的结果，觉得不满意就重来——适合对质量要求高的场景，比如写文章、做方案。

我们常见的 Agent 比如豆包 App、元宝 App 等，还有编程 Agent 像 Claude Code、Codex、Trae 等。

## 07 给他操作手册——按规矩办事

回到正题，邮件回复完了，你抽空去检查了一下，差点被气到——很多邮件回复得乱七八糟，根本不像你自己在回，对于重要的事情他直接帮你做了决定也没征求你同意。你决定再教他做人！

你把自己平时是怎么回邮件的整个过程和思路写成了一份详细的文档：邮件分成哪几类、每种类型应该怎么回、什么情况下必须先征求你的意见。文档写好了之后告诉他：以后回邮件就按这个文档来，这是你的操作手册。

这份操作手册是怎么生效的呢？其实每次跟他开始对话之前，这份手册就已经被悄悄塞进了他的「大脑」里，作为这次对话中必须遵循的行为准则——这叫系统提示（System Prompt），你可以理解为给他注入了一个「底层性格」和「行为规范」。他所有的回答，都会在这个框架之内进行，不会越界。

果然，有了操作手册之后，回复的质量高了不少，你也放心了很多。这份操作手册就是我们常说的 Skills，大模型的行动指南。

> Skills + System Prompt：Skills 是大模型的操作手册，定义了它在特定场景下应该怎么做。System Prompt 是注入到每次对话开始前的底层行为规范。两者配合，让大模型「按你的规矩来，不越界」。

## 08 一个人搞不定？组团队！

到这里，你已经把一个光秃秃的大脑改造成了一个相当靠谱的助手——他能看能听，有记忆，会干活，还能按规矩办事。但是，你很快又遇到了新问题：他虽然什么都能做一点，但什么都不是特别精。让他回邮件，能回，但速度和质量跟专门的客服比还有差距；让他做 PPT，能做，但排版审美总是差点意思；让他分析数据，也行，但深度不够。一个人又当客服、又当设计师、又当数据分析师，再厉害也忙不过来。

你灵机一动：既然一个人搞不定，那就招一个团队嘛！于是你又「招聘」了几个大脑，给每个大脑分别装上不同的零件、配上不同的工具箱、写入不同的操作手册——一个专门负责回邮件，一个专门负责做 PPT，一个专门负责分析数据。然后让它们分工协作：收到邮件的时候由邮件专员处理，遇到需要做汇报的，邮件专员把任务转给 PPT 专员，PPT 专员需要数据的时候再去问数据专员。每个 Agent 各司其职，专业的人做专业的事。

> 多智能体协作 Multi-Agent：单个 Agent 是全能型选手，什么都能做但不够精；多个 Agent 组成团队，每个都有自己的专长，互相配合，就能完成更复杂的任务。现在很多先进的 AI 系统背后就是一群 Agent 在协同工作——你看到的可能只是一个对话窗口，但后面可能有好几个「人」在忙碌。

## 全篇总结：改造大脑全记录

| 改造 | 对应能力 | 他能做什么 |
|---|---|---|
| 原始大脑 | 大模型本体 | 接受文字、思考、输出文字 |
| 装上眼睛 | 视觉理解 | 看懂图片和视频 |
| 装上耳朵 | 语音理解 | 听懂你说的话 |
| 装上记忆 | 上下文窗口 + 知识库 | 记住当前对话，检索历史知识 |
| 装上双手和工具箱 | Function Calling + API / MCP | 调用外部工具，真正干活 |
| 教他拆任务 | Agent 框架 | 自主规划多步骤任务并执行 |
| 给他操作手册 | Skills + System Prompt | 按照你的规范做事，不越界 |
| 组建团队 | 多智能体协作 | 多个专业 Agent 分工协作 |

## 两个必知概念

### 1. Token = AI 的计价单位

聊越多越贵。Token 就是大模型世界的「计价单位」，聊天越长、记忆越多，消耗的 Token 就越多，花费也就越高。

### 2. 幻觉 = AI 不会说「我不知道」

它只会编一个看起来合理的答案。所以才需要 RAG 来兜底，让它有据可查，而不是凭空编造。

每一次改造，都让这个大脑离「真正的助手」更近一步。而你所做的这一切——从挑选大脑、到装备零件、到制定规范、到组建团队——就是当下千行百业都在做的事情：基于大模型，打造属于自己的 AI 应用。

下一次你用任何 AI 产品的时候，不妨想一想：它装了哪些「零件」？它又是怎么被改造的？你会对它有全新的理解。
